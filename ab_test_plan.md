# План A/B-тестирования
## Цель
Цель A/B-теста — сравнить текущую модель `v1` и новую модель `v2` для задачи прогнозирования дефолта по кредитным картам и понять, стала ли новая версия лучше.
В качестве текущей модели используется `credit_default_lr.joblib`.
## Постановка теста
### Группы
- **Control group (A)** — модель `v1`
- **Test group (B)** — модель `v2`
### Разделение трафика
Трафик клиентов делится случайно в пропорции **50/50**.  
Желательно, чтобы один и тот же клиент всегда попадал в одну и ту же группу, например по `client_id`.
### Продолжительность
Тест лучше проводить **2–4 недели** или до набора достаточного количества данных для сравнения метрик.
## Метрики
### Основная метрика
- **F1-score** для класса дефолта (`1`)
Эта метрика важна, потому что в задаче есть дисбаланс классов, и Accuracy может не отражать реальное качество модели.
### Дополнительная метрика
- **Precision**
Precision важен, потому что банк не хочет слишком часто ошибочно относить нормальных клиентов к рискованным.
## Анализ результатов
Для сравнения моделей можно использовать:
- **bootstrap** для оценки доверительных интервалов для F1-score;
- **z-test** или сравнение долей — если нужно проверить Precision или Recall.
Для каждой метрики строится **95% доверительный интервал**.  
Если у модели `v2` интервал по F1-score лучше, чем у `v1`, и при этом Precision не ухудшается сильно, тест можно считать успешным.
## Критерий успеха
Тест считается успешным, если:
1. `v2` показывает лучший F1-score.
2. Precision не становится заметно хуже.
3. Бизнес-метрика, например ожидаемые потери, не увеличивается.
## Что нужно логировать
Чтобы потом анализировать A/B‑тест, нужно сохранять:
- версию модели;
- признаки клиента;
- предсказание;
- вероятность дефолта;
- группу A или B;
- время ответа сервиса.
## Вывод
A/B‑тест нужен, чтобы проверить, действительно ли новая модель лучше старой не только по метрикам, но и по бизнес-эффекту.  
Это помогает безопасно переводить модель в production.
