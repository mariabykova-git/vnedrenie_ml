# План A/B-тестирования модели прогнозирования дефолта

## Цель теста

Цель A/B-теста — сравнить качество текущей модели `v1` и новой модели `v2` для задачи прогнозирования дефолта по кредитным картам и определить, дает ли новая версия модели улучшение с точки зрения качества предсказаний и бизнес-эффекта.

## Постановка эксперимента

### Группы
- **Control group (A)** — текущая модель `v1`
- **Test group (B)** — новая модель `v2`

### Разделение трафика
Трафик клиентов распределяется случайным образом в соотношении **50/50** между группами A и B.  
Разделение должно быть стабильным для одного клиента, например по hash от `client_id`, чтобы один и тот же клиент не попадал в разные группы при повторных запросах.

### Продолжительность теста
Тест проводится в течение достаточного времени для накопления статистически значимой выборки.  
Минимальная рекомендуемая длительность — **2–4 недели**, либо до достижения заранее рассчитанного объема данных.

## Метрики

### Основная метрика
- **F1-score для класса дефолта (class = 1)**

F1-score выбран как основная метрика, потому что задача является несбалансированной, а эта метрика учитывает одновременно Precision и Recall.

### Дополнительная метрика
- **Precision** для класса дефолта

Precision важен в кредитном скоринге, потому что ложноположительное срабатывание может привести к необоснованному отказу клиенту или к излишнему ужесточению кредитной политики.  
Дополнительно можно отслеживать Recall, если для бизнеса критично не пропускать рискованных клиентов.

## Метод статистического анализа

### Проверка гипотез
- **H0**: различий между моделями `v1` и `v2` по выбранной метрике нет.
- **H1**: модель `v2` статистически значимо лучше модели `v1`.

### Статистический тест
Для сравнения метрик можно использовать:
- **z-test** — если сравниваются долевые метрики;
- **t-test** — если метрика агрегируется по клиентам или по временным окнам;
- **bootstrap** — для F1-score, так как это составная метрика и ее распределение не всегда близко к нормальному.

Для данного проекта наиболее уместен **bootstrap** для F1-score и доверительных интервалов, а также z-test для долей при анализе Precision/Recall.

### Доверительные интервалы
Для каждой метрики строятся **95% доверительные интервалы**.  
Если доверительный интервал разности метрик между `v2` и `v1` не содержит ноль и целевая метрика у `v2` выше, это считается положительным результатом.

## Критерий успешности теста

Тест считается успешным, если выполняются все условия:

1. `v2` показывает статистически значимое улучшение по основной метрике `F1-score`.
2. Дополнительная метрика `Precision` не ухудшается ниже допустимого бизнес-порога.
3. Бизнес-метрика, например ожидаемые финансовые потери, не увеличивается.

## Связь с архитектурой сервиса

Для A/B-теста сервис должен уметь:
- маршрутизировать запросы между версиями `v1` и `v2`;
- сохранять `model_v1.pkl` и `model_v2.pkl`;
- логировать версию модели, которая обслужила запрос;
- собирать результаты предсказаний для последующего анализа.

В более зрелой архитектуре это можно реализовать через feature flag, routing layer или отдельный сервис оркестрации экспериментов.

## Вывод

A/B-тест позволяет проверить, действительно ли новая модель `v2` лучше текущей `v1` не только по ML-метрикам, но и по бизнес-эффекту.  
Это важный этап перед полным переводом новой модели в production.
